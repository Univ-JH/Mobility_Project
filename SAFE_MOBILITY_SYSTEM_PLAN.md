# 이동 장치 안전 주행 및 헬멧 착용 감지 시스템 기획서 (구현 전)

## 1) 프로젝트 목표와 핵심 문제 정의

본 프로젝트는 2륜 개인형 이동장치(전동킥보드, 자전거 등)에서 발생하는 주요 사고 원인을 줄이기 위해, **헬멧 착용 강제 + 위험 주행 감지 + 주행 환경(인도/차도) 기반 자동 제어 + 사용자 모니터링**을 통합한 시스템을 구축하는 것을 목표로 한다.

핵심 문제:
- 헬멧 미착용 상태로 주행 시작
- 급가속/급정거, 넘어짐 등 사고 위험 행동의 실시간 감지 미흡
- 인도/차도 구분 없이 동일한 속도로 주행하는 위험 상황
- 사고/응급 발생 시 빠른 상태 인지와 기록 부재
- H/W(헬멧, 센서, 브레이크)와 S/W(서버, DB, 앱) 분리 운영으로 인한 통합성 부족

핵심 가치:
- **사고 예방(Prevention)**: 위험 상황 이전에 제한/제동
- **즉시 대응(Response)**: 응급 의심 시 빠른 알림 및 상태 보고
- **사후 분석(Analytics)**: 주행 메타데이터를 통한 정책 개선

## 2) 요구사항 정리 (주어진 기술 항목 기반)

### 기능 요구사항
1. 헬멧 압력 센서로 착용 여부 판단, 착용 시에만 주행 가능
2. 헬멧 기울기 센서로 사고/응급 상황 판단
3. 가속도 센서로 급가속/급정거 판단
4. 라즈베리파이 + 하향 카메라 + AI 가속기로 인도/차도 판단
5. 서보 모터로 브레이크를 직접 조작해 자동 감속/제동
6. 헬멧(Arduino Nano 33 BLE) ↔ 이동장치(Raspberry Pi 5) 블루투스 통신
7. 라즈베리파이에서 메타데이터를 DB 저장
8. 웹/웹앱으로 상태 확인, 제어, 기록 조회

### 비기능 요구사항
- 안전성: 오작동 시 Fail-Safe 동작(가속 금지 또는 감속 우선)
- 지연시간: 위험 이벤트 감지~제동 명령 반응 시간 최소화
- 신뢰성: BLE 일시 끊김, 센서 노이즈, 카메라 오판단 대응
- 확장성: 다양한 이동장치 모델과 센서 추가 가능 구조
- 보안성: 장치 인증, 통신 암호화, 민감 데이터 보호

## 3) 시스템 전체 아키텍처 (H/W + S/W 통합)

## 3.1 물리 아키텍처
- **헬멧 노드 (Edge-1)**
  - MCU: Arduino Nano 33 BLE
  - 센서: 압력 센서(착용), IMU/기울기(자세/사고), 필요 시 배터리 상태 센서
  - 역할: 센서 수집/1차 필터링/이벤트 생성/BLE 송신

- **이동장치 노드 (Edge-2)**
  - SBC: Raspberry Pi 5
  - 센서/액추에이터: 하향 카메라, 서보 모터(브레이크), 필요 시 속도 센서
  - AI 가속기: NPU/TPU 계열(Hailo)로 실시간 추론
  - 역할: 다중 센서 융합 판단, 제동 제어, 데이터 업로드

- **클라우드/서버 노드**
  - FastAPI + MQTT 브로커 + MongoDB
  - 웹(React), 모바일(React Native) 클라이언트
  - 역할: 장치 관리, 이벤트 저장/조회, 알림, 운영 대시보드

## 3.2 논리 아키텍처 (계층)
- Sensing Layer: 압력/IMU/가속도/카메라 입력
- Decision Layer: 위험도 계산, 규칙 기반 + AI 결과 융합
- Control Layer: 속도 제한/브레이크 제어(서보)
- Communication Layer: BLE(근거리), MQTT/HTTP(광역)
- Application Layer: 모니터링, 이력 분석, 정책 설정

## 4) 핵심 기술 설계 상세

## 4.1 헬멧 착용 판단 (압력 센서)
- 측정 방식:
  - 헬멧 내부 패드 위치에 압력 센서 배치 (좌/우 또는 전/후 2점 이상 권장)
  - 단일값 임계치보다 **다점 + 시간 유지 조건**으로 오탐 방지
- 판단 로직(예시):
  - `착용 = (압력합 > T1) AND (분산 < T2) AND (유지시간 > 2초)`
  - 탈착 이벤트는 히스테리시스 적용(`T_off < T_on`)
- 안전 정책:
  - 착용 불충족 시 구동 금지(스로틀 제한/모터 차단)
  - 주행 중 탈착 감지 시 단계 감속 후 정지

## 4.2 사고/응급 판단 (기울기/IMU)
- 판단 신호:
  - 급격한 자세 변화(롤/피치)
  - 일정 시간 이상 비정상 자세 유지(예: 누운 상태)
  - 충격 지표(가속도 피크)와 결합해 신뢰도 향상
- 예시 규칙:
  - `충격피크 > A1` AND `기울기 > G1` AND `지속시간 > D1` => 사고 의심
  - 사고 의심 후 `N초` 내 사용자 취소 입력 없으면 응급 이벤트 확정
- 추가 고려:
  - 정차 후 헬멧을 내려놓는 상황과 실제 사고 구분(속도/위치 컨텍스트 결합)

## 4.3 급가속/급정거 판단 (가속도 센서)
- 센서 처리:
  - 저역통과 필터로 노면 진동/노이즈 완화
  - 중력 성분 분리 또는 자세 보정
- 이벤트 정의(초기값 예시):
  - 급가속: `a_forward > +a_thr` for `t > t_min`
  - 급정거: `a_forward < -b_thr` for `t > t_min`
- 활용:
  - 실시간 경고
  - 반복 발생 시 자동 제한 모드 전환
  - 사용자 행동 리포트 지표로 저장

## 4.4 인도/차도 판단 (카메라 + AI 가속기)
- 입력:
  - 하향 카메라 영상(프레임/해상도는 성능과 지연 균형 고려)
- 추론:
  - 경량 분류/세그멘테이션 모델(인도/차도/불확실)
  - 프레임 단건 판단 대신 **시간 윈도우 다수결** 사용
- 출력 정책:
  - 인도 확률 높음: 제한 속도 강화 또는 자동 감속
  - 차도 확률 높음: 기본 속도 정책 적용
  - 불확실/야간/우천: 보수적 속도 제한
- 모델 운영:
  - 데이터셋 수집(주간/야간/우천/노면 재질 다양화)
  - 현장 데이터로 지속 재학습/임계치 보정 계획 포함

## 4.5 자동 제동/속도 제어 (서보 모터)
- 제어 구조:
  - 명령 단계: 정상 -> 경고 감속 -> 강제 감속 -> 비상 제동
  - 서보 각도와 브레이크 압력 매핑 테이블 보정
- 안전 설계:
  - 급제동만 고집하지 않고 단계 감속 우선
  - 서보 고착/통신 실패 시 Fail-Safe (동력 제한 + 사용자 경고)
- 기구적 고려:
  - 브레이크 레버 장력, 반복 피로, 기계적 백래시 보정
  - 수동 브레이크 우선권 보장

## 4.6 BLE 통신 (헬멧 Arduino ↔ Pi)
- 데이터 채널:
  - 실시간 텔레메트리(압력, IMU, 배터리)
  - 이벤트 채널(착용/탈착, 충격, 응급 의심)
  - 하트비트(연결 상태 감시)
- 통신 정책:
  - 주기 송신 + 이벤트 즉시 송신 혼합
  - 패킷 손실 대비 시퀀스 번호/재전송 최소 설계
  - 장치 페어링/인증 키 관리
- 장애 처리:
  - BLE 단절 시 일정 시간 내 재연결 시도
  - 재연결 실패 시 보수적 주행 모드(속도 제한 또는 정지)

## 4.7 데이터 저장 (MongoDB)
- 저장 대상:
  - 세션(주행 시작/종료), 센서 요약, 이벤트 로그, 제동 이력
  - 장치 상태(BLE 연결, 배터리, 모델 버전)
  - 사용자/장치 매핑, 정책 설정
- 모델링 원칙:
  - 실시간 조회용 컬렉션 + 장기 분석용 집계 컬렉션 분리
  - 시계열 인덱스(시간, 장치ID) 필수
  - 원시 데이터 전량 저장 vs 샘플링/요약 저장 전략 병행

## 4.8 서비스/클라이언트 (FastAPI, React, React Native)
- FastAPI:
  - 장치 등록/인증, 사용자 인증, 정책 관리 API
  - 이벤트 조회, 주행 리포트, 응급 알림 API
- MQTT:
  - Edge ↔ Server 비동기 이벤트 전달
  - 토픽 설계: `device/{id}/telemetry`, `device/{id}/event`, `device/{id}/control`
- React/React Native:
  - 실시간 상태 대시보드(착용 상태, 주행 모드, 위험 알림)
  - 주행 이력(지도/시간대/위험 횟수)
  - 장치 설정(속도 정책, 알림 수신, 비상 연락처)

## 5) 안전 제어 상태 기계 (State Machine) 제안

핵심 상태:
- `IDLE` (대기)
- `READY` (헬멧 착용 확인, 출발 가능)
- `RUNNING_NORMAL` (정상 주행)
- `RUNNING_LIMITED` (속도 제한 주행)
- `AUTO_BRAKING` (자동 감속/제동)
- `EMERGENCY` (사고/응급 처리)
- `FAULT` (센서/통신 장애)

주요 전이 예시:
- `IDLE -> READY`: 헬멧 착용 + 통신 정상
- `READY -> RUNNING_NORMAL`: 출발 조건 충족
- `RUNNING_NORMAL -> RUNNING_LIMITED`: 인도 감지/반복 난폭 주행
- `RUNNING_* -> AUTO_BRAKING`: 고위험 이벤트 발생
- `AUTO_BRAKING -> EMERGENCY`: 충격 + 비정상 자세 지속
- `ANY -> FAULT`: BLE 단절/센서 이상/서보 불능
- `FAULT -> IDLE`: 복구 및 사용자 재인증

설계 원칙:
- 불확실하면 보수적으로 제한
- 자동 제어보다 사용자 안전 경고를 먼저 병행
- 복구 시에도 즉시 정상모드 복귀 대신 확인 단계 유지

## 6) 데이터 스키마 초안 (MongoDB)

### collections 제안
- `users`
  - 사용자 계정, 권한, 비상 연락처
- `devices`
  - 장치ID, 펌웨어 버전, 센서 구성, 마지막 접속
- `rides`
  - 주행 세션 정보(시작/종료, 거리, 평균 속도, 이벤트 요약)
- `events`
  - 이벤트 타입(helmet_off, harsh_brake, sidewalk_detected, emergency), 시각, 신뢰도
- `telemetry_summary`
  - 1초/5초 단위 요약 통계(가속도 RMS, 기울기 평균, 추론 확률 등)
- `control_logs`
  - 자동 제동/감속 명령 기록 및 결과

인덱스 제안:
- `events`: `{deviceId:1, timestamp:-1}`
- `rides`: `{userId:1, startTime:-1}`
- `telemetry_summary`: `{deviceId:1, bucketTime:-1}`

## 8) 성능/안전 목표 지표 (초기 KPI)

- 헬멧 착용 판정 정확도: 98% 이상(실사용 환경)
- 인도/차도 분류 정확도: 95% 이상(주/야간 통합)
- 위험 이벤트 감지 지연: 300ms~1s 이내(이벤트 종류별 구분)
- 자동 제동 명령 반응: 500ms 이내 목표(기구/제어 포함)
- BLE 재연결 성공률: 99% 이상(단절 후 지정 시간 내)
- 시스템 가용성: 99% 이상(서버 + 장치 통합)

## 9) 개발 단계 계획 (구현 전 로드맵)

### Phase 0: 요구사항 정교화/안전 규격 정의
- 주행 가능/불가 조건 명문화
- 법규 및 안전 규정 확인(인도 주행 정책, 경고 방식 등)
- 실패 모드와 책임 경계 정의

### Phase 1: 헬멧 노드 프로토타입
- 압력/IMU 센서 데이터 취득 및 필터링

- BLE 텔레메트리 송신 안정화

### Phase 2: 이동장치 노드 프로토타입
- Pi 카메라 파이프라인 구축

- 서보 브레이크 제어 캘리브레이션

### Phase 3: 통합 제어/통신
- 헬멧-이동장치 BLE 연동

- MQTT/FastAPI 연결 및 이벤트 업로드

### Phase 4: 서비스/앱 구현
- 관리자/사용자 대시보드
- 주행 이력/이벤트 조회 및 정책 설정

### Phase 5: 현장 테스트/튜닝
- 다양한 도로/조도/날씨 테스트
- 오탐/미탐 분석 후 임계치 및 모델 보정
- 반복 주행으로 신뢰성/내구성 검증

- 센서 오탐/미탐

- 영상 모델 도메인 편향

- 기계적 브레이크 마모/오정렬
  - 대응: 정기 캘리브레이션, 진단 로그, 수동 우선권 보장
- 개인정보/위치 데이터 보안

## 11) 테스트 전략 (구현 전 정의)

### 단위 테스트
- 센서 필터링, 임계치 판정, 상태 전이 로직

### 통합 테스트
- BLE 연결/재연결, 이벤트 전달, 제동 명령 경로

### 시나리오 테스트
- 헬멧 미착용 출발 시도
- 주행 중 헬멧 탈착
- 급가속/급정거 반복
- 인도 진입 감지 후 자동 제한
- 충격 발생 후 응급 시나리오

### 현장 실험
- 노면 종류(아스팔트/보도블록), 시간대(주간/야간), 날씨 조건(맑음/비)

## 12) 구현 전 결정이 필요한 핵심 쟁점

1. 자동 제동 강도 정책: 위험도 기반 연속 제어 vs 단계형 제어
2. 인도/차도 AI 모델: 분류 모델 단순화 vs 세그멘테이션 고정밀
3. Edge-Cloud 역할 분담: 로컬 실시간 판단 비율과 서버 의존도
4. 데이터 저장 정책: 원시 데이터 보관 기간과 요약 데이터 중심 운영
5. 응급 상황 UX: 사용자 취소 시간, 연락 체계, 오탐 대응 절차

---

## 13) 권장 초기 MVP 범위

MVP는 아래를 우선 완성:
- 헬멧 착용 강제(출발 차단)
- 급가속/급정거 감지 + 경고
- 인도/차도 2분류 기반 제한 속도
- 기본 자동 감속(서보 1단계)
- 주행 이벤트 저장 + 웹 대시보드 조회

MVP 이후 확장:
- 사고 확률 점수화 모델
- 앱 기반 실시간 푸시 알림
- 장치 원격 펌웨어 관리(OTA)와 정책 A/B 테스트

---

## 14) 최종 정리

본 시스템의 핵심은 단순 센서 수집이 아니라, **센서 융합 + 실시간 안전 제어 + 서비스 운영 데이터화**를 하나의 흐름으로 묶는 것이다.  
구현 전 단계에서 가장 중요한 것은 임계치 자체보다, **오탐 시 안전하게 완화하고 미탐 시 위험을 즉시 제어하는 상태 기계 설계**이며, 이후 현장 데이터 기반 튜닝을 전제로 개발하는 것이 현실적이다.
