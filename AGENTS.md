# Mobility Project – 개발 규칙 (요약)

이 문서는 팀/AI/코드리뷰가 **같은 기준**으로 개발하기 위한 요약 규칙이다.  
상세 코딩 규칙은 `.cursor/rules/`의 항목을 따른다.

## 공통 원칙
- **안전 우선(Fail-Safe)**: 불확실하면 감속/정지 또는 기능 제한이 기본값이다.
- **재현 가능성**: 실험/튜닝 값(임계치, 모델 버전, 정책)은 모두 설정 파일/DB에 남긴다.
- **관측 가능성**: 모든 주요 판단(착용, 인도/차도, 제동)은 로그/이벤트로 추적 가능해야 한다.
- **보안 기본값**: 장치 인증/권한/비밀키는 코드에 하드코딩하지 않는다.

## 브랜치/PR 규칙
- **브랜치**: `feature/<topic>`, `fix/<topic>`, `chore/<topic>`, `docs/<topic>`
- **PR 크기**: 한 PR은 한 목적(기능/수정/리팩터링)으로 제한
- **필수 포함**:
  - 변경 요약(왜 필요한지)
  - 테스트 방법(로컬/장치/시뮬레이션)
  - 안전 영향(제동/제한/응급 로직 변경 시)

## 커밋 메시지
- 형식: `<type>: <subject>`
  - `feat`, `fix`, `refactor`, `docs`, `test`, `chore`
- 예: `feat: add sidewalk classification event pipeline`

## 테스트/검증(최소)
- 서버(FastAPI): 핵심 로직 단위 테스트(정책/권한/이벤트 수집)
- 프론트: 주요 화면 렌더/상태 전이 스모크 테스트
- 엣지: 센서 입력 단절/노이즈/재연결 시나리오 테스트(로그로 확인)

## 민감 정보
- `.env`, 인증키, 토큰, DB 비밀번호는 커밋 금지
- 샘플은 `.env.example`로 제공
